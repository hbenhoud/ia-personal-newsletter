# AI Personal Newsletter

A self-hosted Go CLI that automatically generates a personal AI-focused newsletter. It fetches RSS feeds, filters articles by semantic similarity to your profile using free embedding APIs, summarizes them with a free LLM, and renders a static HTML site you can open locally or deploy to GitHub Pages.

## Prerequisites

- **Go 1.24+** — [install](https://go.dev/dl/)
- At least one free API key:
  - **Groq** (LLM, free tier — recommended) — [console.groq.com](https://console.groq.com)
  - **Gemini** (embeddings, free tier) — [aistudio.google.com](https://aistudio.google.com)
  - **HuggingFace** (embeddings, free tier) — [huggingface.co/settings/tokens](https://huggingface.co/settings/tokens)

Set environment variables before running (or put them in a `.env` file at the project root):

```bash
export GROQ_API_KEY=gsk_...
export GEMINI_API_KEY=AIza...
# or for HuggingFace embeddings:
export HUGGINGFACE_API_KEY=hf_...
```

## Installation

```bash
git clone https://github.com/hbenhoud/ia-personal-newsletter.git
cd ia-personal-newsletter
go build -o newsletter ./cmd/newsletter
```

This produces a `newsletter` binary in the current directory. Optionally move it to your `$PATH`:

```bash
mv newsletter /usr/local/bin/newsletter
```

## Quick Start

```bash
# 1. Set up a profile (interactive wizard) — one per newsletter edition
newsletter profile setup --profile technical
newsletter profile setup --profile business
#    then add each profile (with its RSS sources) under `profiles:` in config/newsletter.yaml

# 2. Generate all newsletters (or one with --profile <name>)
newsletter generate

# 3. Open output/index.html — the landing page links to every edition
```

## Configuration

All configuration lives in the `config/` directory.

### newsletter.yaml

Holds the `profiles:` list (one newsletter edition each, with its own RSS
sources and `profile.md`) plus the shared filtering, embedding, LLM, and output
settings.

```yaml
# One entry per newsletter edition. Each profile has its own RSS sources and its
# own profile.md (interests, theme, language, level, recency window).
profiles:
  - name: technical
    profile: profiles/technical.md
    sources:
      rss:
        - https://simonwillison.net/atom/everything/
        - https://huggingface.co/blog/feed.xml
        - https://blog.langchain.dev/rss.xml
        - https://hnrss.org/frontpage
  - name: business
    profile: profiles/business.md
    sources:
      rss:
        - https://venturebeat.com/category/ai/feed/
        - https://techcrunch.com/category/artificial-intelligence/feed/

# (Backward-compat: a top-level `sources:` block with no `profiles:` is treated
# as a single "default" profile pointing at config/profile.md.)

filtering:
  mode: semantic              # semantic | keyword | both
  similarity_threshold: 0.72  # 0.0–1.0; lower = more articles, higher = stricter
  max_articles: 20            # maximum articles passed to LLM summarization
  exclude_keywords: []        # hard exclusions applied before embedding (e.g. [crypto, NFT])

embedding:
  provider: gemini            # gemini | huggingface
  model: gemini-embedding-001 # current Gemini embedding model (v1beta)
  api_key_env: GEMINI_API_KEY # name of the env var holding the API key
  cache_path: ./.cache/embeddings.json

llm:
  provider: groq              # groq | gemini
  model: llama-3.3-70b-versatile
  api_key_env: GROQ_API_KEY

output:
  site_dir: ./output
  items_per_issue: 10         # final number of articles rendered in the newsletter
```

### config/profiles/&lt;slug&gt;.md

One file per profile, generated by `newsletter profile setup --profile <name>`.
Edit individual fields with `newsletter profile edit --profile <name>`. These
files hold personal data and are gitignored (injected from secrets in CI).

```markdown
# Technical AI profile

I am an expert-level backend developer. I am interested in open source LLMs,
RAG, autonomous agents, local inference tooling, and NLP research papers.

## Topics to avoid
funding rounds, marketing announcements, AI art

## Preferences
language: fr          # en | fr — language for the newsletter output
recency_days: 3       # only articles published in the last N days
theme: dark           # minimal | dark | paper | terminal
level: expert         # beginner | intermediate | expert
```

**How each profile is used:**
The full text of a profile's `.md` is embedded into a vector that serves as its
reference point. Every article from that profile's sources is embedded too, and
only articles whose vector is close enough (above `similarity_threshold`) are
kept. The `Interests` and `Level` fields are also injected into the LLM
summarization prompt to personalise the output. Each profile is rendered
independently into `output/<slug>/`.

> **Profile tip:** Use specific technical vocabulary (RAG, LoRA, llama.cpp, fine-tuning…) rather than generic terms — it significantly improves semantic matching quality.

## Architecture

```
ia-personal-newsletter/
├── cmd/newsletter/main.go         CLI entry point — parses args, wires components
├── internal/
│   ├── config/config.go           Loads newsletter.yaml + profile.md, runs wizards
│   ├── ingestion/rss.go           Concurrent RSS fetch, Article normalisation, URL dedup
│   ├── filtering/semantic.go      3-pass filter: recency → exclusion → cosine similarity
│   ├── embedding/
│   │   ├── embedder.go            Embedder interface
│   │   ├── gemini.go              Gemini gemini-embedding-001 implementation (v1beta)
│   │   ├── huggingface.go         HuggingFace Inference API implementation
│   │   ├── cache.go               SHA256-keyed JSON cache (load on start, flush on exit)
│   │   └── factory.go             Creates the configured Embedder
│   ├── llm/
│   │   ├── provider.go            Provider interface + GenerationConfig
│   │   ├── groq.go                Groq (OpenAI-compatible) implementation
│   │   ├── gemini.go              Gemini generateContent implementation
│   │   └── factory.go             Creates the configured Provider
│   ├── generation/summarizer.go   Renders prompt template, calls LLM, parses TL;DR output
│   └── site/generator.go          Writes output/<slug>/YYYY-MM-DD/index.html + per-profile index; WriteHome writes output/index.html
├── config/
│   ├── newsletter.yaml            User-editable configuration (profiles list + shared settings)
│   └── profiles/<slug>.md         One per profile, generated by wizard (gitignored)
├── templates/
│   ├── newsletter.html            Issue page template
│   ├── index.html                 Per-profile issues list template
│   ├── home.html                  Root landing page listing all editions
│   ├── prompts/summarize.tmpl     LLM prompt template
│   └── themes/
│       ├── minimal.css            Editorial card layout, serif + sans-serif
│       ├── dark.css               Premium dark mode, indigo accents
│       ├── paper.css              Newspaper masthead, Palatino serif
│       └── terminal.css           Retro CRT, amber palette, monospace
├── .cache/embeddings.json         Persistent embedding cache, shared across profiles (auto-generated)
└── output/                        Generated static site
    ├── index.html                 Landing page linking to every edition
    └── <slug>/                    One directory per profile
        ├── index.html             That profile's issues list
        └── YYYY-MM-DD/index.html  A dated issue
```

## Execution Pipeline

```
newsletter generate            (loops steps 2–7 once per profile)
       │
       ▼
[1] Load config
    config/newsletter.yaml (profiles list) + each config/profiles/<slug>.md
    Build shared embedder + LLM provider once
       │
       ▼
[2] Ingest RSS feeds (concurrent, this profile's sources)
    github.com/mmcdole/gofeed
    → []Article{Title, URL, Content, Source, Published}
       │
       ▼
[3] Pre-filter (no API calls)
    ├─ Drop articles older than recency_days
    ├─ Drop articles matching exclude_keywords
    └─ Deduplicate by URL
       │
       ▼
[4] Embed articles + profile
    ├─ Load embedding cache (.cache/embeddings.json)
    ├─ Embed profile.md text → reference vector
    └─ Embed each article (title + description, capped at 1500 chars)
       │  (cached vectors skip API call)
       ▼
[5] Score + rank
    ├─ Cosine similarity(article_vec, profile_vec) for each article
    ├─ Keep articles above similarity_threshold
    └─ Sort descending, keep top max_articles
       │
       ▼
[6] Summarize (LLM)
    For each selected article:
    └─ Render summarize.tmpl → call LLM → parse TL;DR + Why it matters
       │
       ▼
[7] Render static site (this profile)
    ├─ Load theme CSS (templates/themes/{theme}.css)
    ├─ Embed CSS inline → output/<slug>/YYYY-MM-DD/index.html
    └─ Regenerate output/<slug>/index.html (that profile's issues list)
       │
       ▼
[8] After all profiles: rebuild output/index.html landing page
       │
       ▼
    Done. Clickable file:// paths printed in the terminal.
```

## Available Themes

Set via `newsletter profile setup` or `newsletter profile edit`.

| Theme | Style |
|---|---|
| `modern` | Stripe-inspired light SaaS UX: blue-gray canvas, white cards, blurple `#635bff` accents, signature angled gradient hero (teal→blurple→violet) |
| `minimal` | White background, Georgia serif body, editorial card layout with blue accent bar |
| `dark` | Near-black background, indigo `#818cf8` accents, Vercel/Linear aesthetic, gradient header |
| `paper` | Cream background, Palatino serif, newspaper masthead with double rules, ornamental dividers |
| `terminal` | Near-black background, amber `#ffb700` palette, CRT scanlines, monospace, blinking cursor |

All themes:
- Embed CSS inline — no internet required to view
- Use a 980px max-width single-column layout
- Are fully responsive down to mobile (600px breakpoint)

## CLI Reference

| Command | Description |
|---|---|
| `newsletter generate` | Generate every configured profile's newsletter |
| `newsletter generate --profile <name>` | Generate only the named profile |
| `newsletter profile list` | List configured profiles (theme, language, level, feed count) |
| `newsletter profile setup --profile <name>` | Interactive wizard to create/overwrite a profile |
| `newsletter profile show --profile <name>` | Print a profile's `.md` |
| `newsletter profile edit --profile <name>` | Re-run the wizard with current values pre-filled (Enter = keep) |
| `newsletter sources list [--profile <name>]` | List RSS feed URLs (all profiles, or one) |
| `newsletter llm test` | Test LLM provider connectivity |
| `newsletter embedding test` | Test embedder connectivity and print vector dimensions |

> `--profile` is optional when a single profile is configured; with multiple profiles, `profile show/edit` require it.

## Adding a New Provider

### New LLM Provider

1. Create `internal/llm/myprovider.go` implementing the `Provider` interface:

```go
type Provider interface {
    Complete(ctx context.Context, prompt string, cfg GenerationConfig) (string, error)
    Name() string
}
```

2. Register it in `internal/llm/factory.go`:

```go
case "myprovider":
    return NewMyProvider(apiKey, model), nil
```

3. Use it: set `llm.provider: myprovider` and `llm.api_key_env: MY_API_KEY` in `newsletter.yaml`.

### New Embedding Provider

1. Create `internal/embedding/myprovider.go` implementing the `Embedder` interface:

```go
type Embedder interface {
    Embed(ctx context.Context, text string) ([]float64, error)
    EmbedBatch(ctx context.Context, texts []string) ([][]float64, error)
}
```

2. Register it in `internal/embedding/factory.go`:

```go
case "myprovider":
    return NewMyEmbedder(apiKey, model, cache), cache, nil
```

3. Use it: set `embedding.provider: myprovider` and `embedding.api_key_env: MY_EMBED_KEY` in `newsletter.yaml`.

The cache is automatically shared — no changes needed to `cache.go`.

> **Note:** When switching embedding models, delete `.cache/embeddings.json` — vector dimensions differ between models and will cause cosine similarity errors.

## Deploying the Site

The `output/` directory is a self-contained static site. Each HTML file embeds all CSS inline.

### GitHub Pages

```bash
git subtree push --prefix output origin gh-pages
```

Or use a GitHub Actions workflow to run `newsletter generate` on a schedule and push the output.

### Netlify Drop

Drag and drop the `output/` folder at [app.netlify.com/drop](https://app.netlify.com/drop).

### Local preview

```bash
cd output && python3 -m http.server 8080
```

## FAQ / Troubleshooting

**`error: environment variable GROQ_API_KEY is not set`**
Run `export GROQ_API_KEY=your_key` before running the CLI, or add it to a `.env` file at the project root.

**`error: reading profile "<name>" … run 'newsletter profile setup --profile <name>' first`**
The profile file referenced in `newsletter.yaml` is missing. Create it with `newsletter profile setup --profile <name>`.

**`No articles matched your profile`**
Your `similarity_threshold` may be too high. Try lowering it in `newsletter.yaml` (e.g. from `0.72` to `0.60`). Also check that your `recency_days` window is wide enough.

**The same articles appear every run**
The embedding cache prevents duplicate API calls but not duplicate articles. To regenerate from scratch, delete `.cache/embeddings.json`.

**`newsletter generate` is slow on first run**
The first run embeds every article (one API call each). Subsequent runs reuse the cache and are much faster.

**HTML output looks unstyled**
Make sure the theme CSS file exists at `templates/themes/{theme}.css`. Run `newsletter profile show --profile <name>` to check which theme is configured.

**I want to add more RSS sources**
Edit `config/newsletter.yaml` and add URLs under the relevant profile's `sources.rss`. Any RSS 2.0 or Atom feed works.

**I want another newsletter edition**
Add a new entry under `profiles:` in `config/newsletter.yaml` (name + sources + profile path), then run `newsletter profile setup --profile <name>` to create its `profile.md`.
